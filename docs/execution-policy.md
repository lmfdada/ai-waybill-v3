# 执行联动策略

审批通过进入执行阶段后，V3 会根据异常类型生成赔付记录和库存流水。下游记录都保留 `approvalId`，可反查触发它们的审批动作。

## 物流异常

| 异常类型 | 赔付方向 | 赔付金额 | 库存动作 |
|---|---|---:|---|
| 丢件 `lost` | 赔付客户 | 工单金额 100% | 重新发货扣减库存 |
| 破损 `damaged` | 赔付客户 | 工单金额 100% | 退货入库 + 重新发货扣减库存 |
| 客户拒收 `rejected` | 无 | 0 | 退货入库 |
| 超时未签收 `timeout` | 赔付客户 | 工单金额 20% | 无库存动作 |
| 地址错误 `address_error` | 无 | 0 | 重新发货扣减库存 |

## 品控异常

| 异常类型 | 赔付方向 | 库存动作 |
|---|---|---|
| 数量不符 `quantity_mismatch` | 向供应商追偿 | 放行/解锁批次 |
| 外观破损 `appearance_damage` | 向供应商追偿 | 退回供应商/退仓 |
| 规格不符 `spec_mismatch` | 向供应商追偿 | 批次作废 + 重采购 |
| 标签错误 `label_error` | 无 | 修正标签后放行 |
| 批次异常 `batch_abnormal` | 向供应商追偿 | 批次作废 + 重采购 |

## 一致性要求

审批、库存、赔付和工单状态变更通过 `runAtomic` 原子工作单元提交：

- SQLite 本地模式使用 `better-sqlite3` transaction 批量写入。
- Neon/Postgres 生产模式使用 `@neondatabase/serverless` 的 `transaction()` 一次提交所有写入。
- 如果持久化失败，服务会回滚内存状态并记录错误，避免出现“审批已完成但库存/赔付未生成”的中间态。
- `approval_records.operationKey` 用于重复点击和任务重试幂等；重复操作不会重复生成审批、赔付或库存流水。
- 赔付记录和库存流水都保存 `approvalId`，可反查是哪一次审批触发。

## 拒绝与重新提交

审批拒绝后，工单进入 `rejected`。上报人或管理员可通过重新提交动作补充说明，系统会根据当前审批规则重新分配一级/二级审批人，并写入一条 `resubmitted` 审计记录。重提同样带 `operationKey`，避免网络重试导致重复流转。

## 批次锁定边界

品控异常锁定粒度为 `skuCode + batchNo`。同一运单同一 SKU 批次重复扫描只追加扫描记录；其他运单扫描同一 SKU 批次时会被阻断并返回当前锁定工单，防止暂扣批次被其他运单继续引用。
