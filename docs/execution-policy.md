# 执行联动策略

审批通过进入执行阶段后，V3 会根据异常类型生成赔付记录和库存流水。下游记录都保留 `approvalId`，可反查触发它们的审批动作。

## 物流异常

| 异常类型 | 赔付方向 | 赔付金额 | 库存动作 |
|---|---|---:|---|
| 丢件 `lost` | 赔付客户 | 工单金额 100% | 重新发货扣减库存 |
| 破损 `damaged` | 赔付客户 | 工单金额 100% | 退货入库 + 重新发货扣减库存 |
| 客户拒收 `rejected` | 无 | 0 | 退货入库 |
| 超时未签收 `timeout` | 赔付客户 | 工单金额 20% | 无库存动作 |
| 地址错误 `address_error` | 无 | 0 | 重新发货扣减库存 |

## 品控异常

| 异常类型 | 赔付方向 | 库存动作 |
|---|---|---|
| 数量不符 `quantity_mismatch` | 向供应商追偿 | 放行/解锁批次 |
| 外观破损 `appearance_damage` | 向供应商追偿 | 退回供应商/退仓 |
| 规格不符 `spec_mismatch` | 向供应商追偿 | 批次作废 + 重采购 |
| 标签错误 `label_error` | 无 | 修正标签后放行 |
| 批次异常 `batch_abnormal` | 向供应商追偿 | 批次作废 + 重采购 |

## 一致性要求

当前本地实现会在同一服务流程内依次写入审批记录、赔付记录、库存流水和工单状态。生产化时建议用数据库事务包裹这些写入，并对 `approval_records.operation_key` 建唯一约束，防止重复点击或任务重试导致重复联动。
