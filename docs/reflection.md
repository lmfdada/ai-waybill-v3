# 反思题

## 1. 如果题目直接给出审批金额阈值和超时时长，架构设计会有什么不同？

不会把阈值写死在代码里。即使产品给了精确数值，也会落在 `approval_rules` 表中，因为阈值很可能随商户、品类、仓库、促销周期变化。差异只是初始种子数据会更确定，假设说明中不需要解释数值来源。

## 2. 审批状态变更与库存/赔付联动之间最大的风险点是什么？不用事务会怎么处理？

最大风险是审批已经显示通过，但赔付或库存流水写入失败，造成业务中间态。优先方案是数据库事务，将审批记录、工单状态、赔付记录、库存流水放在同一事务中提交。

如果不用事务，则采用 outbox/补偿任务：审批通过时先写一条待执行联动任务，任务有幂等键和重试次数；联动成功后再标记任务完成。页面需要展示“执行中/待补偿”，避免误导用户以为已经完成。

## 3. 面对一句话级别模糊需求，是先问清楚还是先做版本？

先识别高风险问题并提出澄清，同时做一个最小可验证版本。对会影响数据模型和状态机的点必须先问，例如是否允许多异常并存、赔付金额是否可改、库存是否接 WMS。对 UI 文案和筛选项这类低风险内容，可以先实现再迭代。

## 4. 新增“客户投诉服务态度”异常类型，表结构和状态机要改什么？

表结构不需要大改，因为异常类型是枚举/配置层扩展。需要增加类型值、展示文案和执行策略。该类型不涉及库存和赔付，可在执行联动策略中配置为“生成客服回访任务/无需库存流水/无需赔付”。状态机仍走同一套上报、审批、执行、完成流程。

## 5. 工单量从 200 到 20 万，最先撑不住的环节是什么？怎么处理？

最先撑不住的是列表查询和定时任务全表扫描。处理方式：

- 工单表增加 `status`、`type`、`assignee_id`、`due_at`、`waybill_no` 索引。
- 列表使用游标分页或稳定排序分页。
- 超时任务只扫描 `status in (...) and due_at <= now()` 的索引范围。
- 接口日志按时间分区或保留最近 N 天。
- 统计数据使用异步聚合表或缓存，不在列表页实时全表 count。

## 6. 给 V2 新增接口时，如何保证不破坏旧调用方？金额字段从 int 改 decimal 怎么办？

新增 `/api/v2/*` 版本化接口，不改现有 `/api/orders/*` 行为。新增字段只追加，不删除、不重命名；废弃字段保留一段兼容期。接口响应使用契约文档和示例固定。

金额字段从 `int` 改为 `decimal` 时，V3 接收层按字符串/decimal 处理，数据库使用 `decimal/numeric`，前端展示时再格式化，避免 JS 浮点误差。V3 还应在接口同步日志中记录字段解析异常，便于灰度期间排查。
